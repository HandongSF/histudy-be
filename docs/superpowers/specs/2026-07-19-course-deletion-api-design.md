# 과목 삭제 API 설계

## 배경

프런트엔드 이슈 [HandongSF/histudy-fe#172](https://github.com/HandongSF/histudy-fe/issues/172)는 방학 중 학생 수요에 따라 추가한 과목이 잘못 입력됐을 때 관리자가 해당 과목을 삭제할 수 있어야 한다는 요구다. 현재 관리자 과목 화면은 현재 학기 과목 조회, 검색, CSV 업로드만 제공한다.

백엔드에는 `POST /api/courses/delete`가 이미 있지만 요청 본문으로 과목 ID를 받고 `1` 또는 `0`을 반환한다. 이 계약은 HTTP 상태만으로 성공과 실패를 구분하기 어렵고, 과거 학기 과목도 삭제할 수 있으며, 신청·그룹·리포트가 참조하는 과목을 삭제하면 데이터베이스 제약 오류가 서버 오류로 노출될 수 있다.

## 목표

- 관리자가 현재 학기에 잘못 등록된 미사용 과목을 명시적인 HTTP 계약으로 삭제할 수 있다.
- 신청, 그룹, 리포트가 사용하는 과목과 과거 학기 데이터는 보존한다.
- 기존 `POST /api/courses/delete` 계약은 호환성을 위해 유지한다.
- 프런트엔드가 상태 코드만으로 삭제 성공과 실패 원인을 처리할 수 있다.

## 비목표

- 과목과 연결된 신청, 그룹, 리포트를 연쇄 삭제하지 않는다.
- 과거 학기 과목을 정리하는 별도 관리자 기능을 만들지 않는다.
- 기존 CSV 과목 교체 동작이나 기존 삭제 엔드포인트의 응답을 변경하지 않는다.
- 프런트엔드 화면은 이 백엔드 PR에서 수정하지 않는다.

## API 계약

### 요청

```http
DELETE /api/courses/{courseId}
Authorization: Bearer <ADMIN token>
```

요청 본문은 없다. `courseId`는 `int64` 경로 파라미터다.

### 응답

| 상태 | 조건 | 본문 |
| --- | --- | --- |
| `204 No Content` | 현재 학기의 미사용 과목을 삭제함 | 없음 |
| `403 Forbidden` | `ADMIN` 권한이 아님 | 기존 `ExceptionResponse` |
| `404 Not Found` | 과목이 없거나 현재 학기 과목이 아님 | 기존 `ExceptionResponse` |
| `409 Conflict` | 신청, 그룹 또는 리포트가 과목을 참조함 | 기존 `ExceptionResponse` |

과거 학기 과목은 관리자 현재 학기 화면의 삭제 대상이 아니며, 역사 데이터의 존재 여부를 API 소비자에게 별도로 노출하지 않도록 `404 Not Found`로 처리한다.

## 서버 구조와 데이터 흐름

1. `CourseController`가 `ADMIN` 권한과 경로 파라미터를 확인한다.
2. `CourseService`가 ID로 과목을 조회한다.
3. 과목이 없거나 `academicTerm.isCurrent`가 아니면 `CourseNotFoundException`을 던진다.
4. `CourseRepository`가 `PreferredCourse`, `GroupCourse`, `StudyCourse` 참조 존재 여부를 조회한다.
5. 참조가 있으면 `CourseInUseException`을 던지고, 없으면 과목을 삭제한다.
6. 컨트롤러는 성공 시 `204 No Content`를 반환한다.

컨트롤러는 권한과 HTTP 응답만 담당하고, 현재 학기 및 사용 여부 결정은 트랜잭션 서비스에 둔다. JPA 구현은 참조 존재 여부라는 조회 형태만 담당한다.

## 오류 처리

`CourseInUseException`을 추가하고 `ExceptionController`의 충돌 응답 대상에 포함한다. 메시지는 관리 화면에서 그대로 안내할 수 있도록 `사용 중인 강의는 삭제할 수 없습니다.`로 고정한다. 예상 가능한 참조 충돌을 일반 `RuntimeException` 처리기로 보내거나 Discord 서버 오류 알림으로 기록하지 않는다.

삭제 직전 다른 트랜잭션이 참조를 생성하는 경쟁 조건은 데이터베이스 FK가 최종 방어선으로 남는다. 이번 PR은 관리자 단일 작업 흐름에 필요한 명시적 선행 검사까지만 다룬다.

## 호환성

기존 `POST /api/courses/delete`와 `CourseIdDto`는 제거하거나 동작을 바꾸지 않는다. 신규 프런트엔드 구현은 `DELETE /api/courses/{courseId}`만 사용한다. OpenAPI 문서에는 기존 경로를 유지하면서 신규 경로를 권장 계약으로 추가한다.

## 테스트

- 컨트롤러 테스트: 관리자 성공 `204`, 일반 사용자 `403`, 서비스 예외의 `404` 및 `409` 응답을 검증한다.
- 서비스 테스트: 현재 학기 미사용 과목 삭제, 없는 과목 거부, 과거 학기 과목 거부, 사용 중 과목 거부를 검증한다.
- 리포지토리 테스트: 신청, 그룹, 리포트 각각의 참조가 사용 중으로 판별되고 참조 없는 과목은 사용 중이 아님을 검증한다.
- 회귀 검증: 기존 `POST /api/courses/delete` 테스트와 전체 테스트 스위트를 유지한다.

## 프런트엔드 연동 메모

관리자 과목 목록의 각 행에서 삭제 확인 후 `DELETE /api/courses/{course.id}`를 호출한다. `204`면 과목 목록 쿼리를 갱신하고, `409`면 응답 메시지를 사용자에게 안내한다. `404`는 이미 삭제됐거나 삭제 대상이 아닌 항목으로 안내한 뒤 목록을 갱신한다.
